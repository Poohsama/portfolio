# EC Sales Aggregation

ECモールから出力された売上CSVを読み込み、商品ごとの販売数を集計し、管理用Excelへ継続的に蓄積する業務改善ツールです。

モールごとに異なるCSV形式を吸収しながら、日次・月次の集計に利用できるデータを、一定の手順で作成することを目的として開発しました。

## 解決したかった業務課題

ECモールの売上データは、モールごとにCSVの形式が異なります。

同じ売上情報であっても、次のような違いがあります。

- 見出し名
- 列の並び
- 商品番号やSKUの持ち方
- 販売数を示す列
- 文字コード
- データの出力単位

そのため、複数モールの売上を集計するには、それぞれのCSVを手作業で確認し、必要な列を探して転記する必要がありました。

また、日々の販売数を継続して蓄積する場合、次のような問題もありました。

- 別のモールのCSVを誤って使用する
- 対象列を間違えて転記する
- 既存データを上書きする
- 貼り付け開始位置を間違える
- 商品番号やSKUの表記差により集計がずれる
- 担当者によって集計手順や結果が変わる

そこで、CSVの内容を判定し、必要なデータを抽出・集計して、管理用ファイルの正しい位置へ追加する仕組みを作成しました。

## 設計方針

このツールでは、特定のモールだけに固定した処理ではなく、将来ほかのモールを追加できる構成を重視しています。

主な設計方針は次のとおりです。

- モールごとのCSV形式を個別に管理する
- 集計処理そのものは可能な範囲で共通化する
- CSVの見出しから対象モールやファイル形式を確認する
- 出力先の既存データを上書きしない
- 日々のデータを継続して追加できるようにする
- モール差分を設定ファイルへ分離する
- 仕様変更時の修正箇所を限定する
- 処理内容と変更箇所が分かるコメントを残す

すべてのモールを一つの処理へ無理に押し込むのではなく、共通部分と個別部分を分けて管理しています。

## 主な機能

- 売上CSVの読み込み
- CSVの文字コード判定
- 必須見出しの確認
- 対象モール・CSV形式の判定
- 商品番号・SKUの抽出
- 商品ごとの販売数集計
- 管理用Excelへのデータ追加
- 既存データ末尾の空き位置判定
- モールごとの出力先切り替え
- 設定ファイルによる差分管理
- エラー内容の表示
- 処理結果のログ出力

## 処理の流れ

1. 設定ファイルを読み込む
2. 対象CSVを読み込む
3. CSVの見出しを確認する
4. 想定したモールのCSVか判定する
5. 必要な列を抽出する
6. 商品番号またはSKU単位で販売数を集計する
7. 出力先Excelを開く
8. 指定列の最終データ位置を確認する
9. その下に新しい集計結果を追加する
10. ファイルを保存する
11. 処理結果をログへ記録する

## CSV形式の判定

ファイル名だけでは、CSVの内容が正しいか判断できません。

そのため、CSV内の見出しを確認し、想定した形式であるかを処理前に判定します。

たとえば、モールごとに必要な見出しを設定し、該当する項目が不足している場合は処理を中止します。

```yaml
malls:
    tiktok:
        required_headers:
            - Seller SKU
            - Quantity
        sku_column: Seller SKU
        quantity_column: Quantity

    amazon:
        required_headers:
            - seller-sku
            - quantity
        sku_column: seller-sku
        quantity_column: quantity
```

実際の見出し名や設定内容は、使用するCSV仕様に合わせて変更します。

この判定により、別用途のCSVや異なるモールのCSVを誤って処理する事故を防ぎます。

## モール差分と共通処理の分離

モールごとにCSVの形式は異なりますが、集計処理には共通する流れがあります。

```text
モール別処理
├─ 文字コード
├─ 必須見出し
├─ SKU列
├─ 販売数列
├─ 除外条件
└─ 出力先設定

共通処理
├─ CSV読み込み
├─ 入力形式の確認
├─ SKUごとの販売数集計
├─ 既存データ末尾の確認
├─ Excelへの追加
└─ ログ出力
```

新しいモールを追加する際は、既存の集計処理を複製するのではなく、モール固有の設定や前処理を追加する形を目指しています。

## 累積データの追加

このツールでは、集計結果を毎回同じセルへ書き込むのではなく、指定列の一番下にある空き位置を確認してから追加します。

```text
既存データ
├─ 1日目の販売数
├─ 2日目の販売数
├─ 3日目の販売数
└─ 次の空き位置 ← 今回の結果を追加
```

これにより、過去の集計結果を上書きせず、日々の販売実績を継続して蓄積できます。

貼り付け開始位置を人が判断する必要がないため、行ずれや上書きの防止にもつながります。

## SKU集計

CSV内に同じSKUが複数行存在する場合は、販売数をまとめてから出力します。

```python
def aggregate_sales(rows: list[dict], sku_column: str, quantity_column: str) -> dict[str, int]:
    """
    CSV内の販売数をSKU単位で集計する。

    モールごとの列名は引数で受け取り、
    集計処理自体は共通で利用する。
    """
    result: dict[str, int] = {}

    for row in rows:
        sku = str(row.get(sku_column, "")).strip()
        quantity = int(row.get(quantity_column, 0) or 0)

        if not sku:
            continue

        result[sku] = result.get(sku, 0) + quantity

    return result
```

実際のコードでは、CSVの値や欠損データに応じた確認処理も行います。

## 誤操作を減らすための工夫

日次集計では、処理そのものよりも入力ファイルや転記位置の間違いが問題になりやすいため、次の対策を行っています。

- 必須見出しの確認
- モールとCSV形式の照合
- 対象ファイルの存在確認
- 空のSKUを除外
- 数量データの確認
- 出力列の指定
- 最終入力位置の自動判定
- 元データを上書きしない
- エラー時は保存せず処理を中止
- 実行内容をログへ記録

処理を自動化するだけでなく、誤った操作を実行できないようにすることも設計の一部と考えています。

## 拡張性

今後、ほかのECモールでも同様の集計を行うことを想定しています。

そのため、新しいモールを追加する場合は、主に次の項目を設定または追加します。

- CSVの文字コード
- 必須見出し
- SKU列
- 販売数列
- モール固有の前処理
- 出力先シート
- 出力先列
- 除外条件

集計、末尾判定、Excelへの追加などの共通処理は再利用します。

これにより、モールが増えるたびに新しいプログラムを一から作るのではなく、既存の仕組みを拡張して対応できます。

## 属人化を減らすための工夫

手作業で集計する場合、担当者はモールごとのCSV構造や転記ルールを覚えておく必要があります。

このツールでは、その判断を設定と処理ルールとして残しています。

- CSVのどの列を使うか
- どの見出しでファイルを判定するか
- どの単位で販売数を集計するか
- どのシートや列へ追加するか
- どの位置から書き込むか

これらを仕組みとして固定することで、担当者の記憶や経験に依存せず、同じ手順と結果を再現できるようにしています。

このツールにより、CSVを開いて対象列を確認し、商品ごとの販売数を集計して、管理用Excelへ手作業で転記する工程を減らすことができました。

また、CSV形式の判定と出力位置の自動確認により、別ファイルの使用、列の選択ミス、既存データの上書きといった事故も起こりにくくなりました。

単に集計時間を短縮するだけではなく、モールごとに異なるデータ形式を共通の運用へ変換し、継続して蓄積できる仕組みにした点が、このツールの中心です。
