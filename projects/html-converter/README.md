# EC HTML Converter

楽天市場向けに作成された商品ページHTMLを、Yahoo!ショッピングやau PAY マーケットなど、別のECモール向けに変換する業務改善ツールです。

ECモールごとに異なるHTML制限、画像URL、商品リンク、使用可能タグなどの差分を整理し、手作業による修正を減らすことを目的として開発しました。

## 解決したかった業務課題

ECモールでは、同じ商品を掲載する場合でも、利用できるHTMLタグや画像URL、リンク形式などが異なります。

そのため、楽天市場向けに作成したHTMLをほかのモールへ転用する際には、次のような手作業が発生していました。

- 使用できないHTMLタグの削除
- 画像URLの置換
- 商品リンクやカテゴリリンクの変更
- PC用・スマートフォン用HTMLの調整
- モール固有画像や装飾の削除
- HTML属性や記述形式の修正

修正箇所が多いため、作業時間がかかるだけでなく、置換漏れや誤ったURLの混入が発生しやすい状態でした。

そこで、楽天市場向けHTMLを元データとして読み込み、各モールの仕様に合わせて自動変換する仕組みを作成しました。

## 設計方針

このツールでは、すべての変換処理を一つの大きな処理にまとめるのではなく、役割ごとに分離しています。

主な設計方針は次のとおりです。

- 共通して使用できる変換処理を分離する
- モール固有の処理を個別に管理する
- PC用とスマートフォン用の差分を切り分ける
- 置換ルールを追加・変更しやすくする
- 変換前後の内容を確認できるようにする
- 将来、新しいモールを追加できる構成にする

コードの短さよりも、仕様変更時に修正箇所を特定しやすいことを重視しています。

## 主な機能

- 楽天市場向けHTMLの読み込み
- Yahoo!ショッピング向けHTMLへの変換
- au PAY マーケット向けHTMLへの変換
- 使用禁止タグの削除
- `font` タグなど不要な装飾タグの除去
- 画像URLの置換
- 商品リンク・カテゴリリンクの置換
- モール固有画像の削除
- PC用・スマートフォン用HTMLの個別変換
- HTML属性や終了タグの調整
- フォルダ内ファイルの一括変換
- 処理結果のログ出力

## 処理の流れ

1. 変換対象のHTMLまたはCSVを読み込む
2. 元データが想定した形式か確認する
3. モール共通の不要要素を削除する
4. 画像URLやリンクを変換する
5. 出力先モール固有の処理を適用する
6. PC用・スマートフォン用の差分を調整する
7. 変換後のHTMLを書き出す
8. 処理結果をログへ記録する

## 共通処理と個別処理の分離

モールごとの変換処理には、共通する部分と個別に対応する部分があります。

たとえば、不要なタグの削除や改行タグの統一は、複数モールで共通して利用できます。

一方で、画像URL、商品リンク、利用可能なタグなどは、モールごとに異なります。

そのため、処理を次のように分けています。

```text
共通処理
├─ 不要タグの削除
├─ HTML記述形式の統一
├─ 不要画像の削除
└─ 共通の文字列整形

モール別処理
├─ Yahoo!ショッピング向け変換
├─ au PAY マーケット向け変換
└─ 将来追加するモール向け変換
```

これにより、共通仕様を変更する場合と、特定モールだけを変更する場合を切り分けられます。

## モール差分への対応

モールごとの差分は、処理内容ごとに整理しています。

主な差分の例は次のとおりです。

| 項目 | 楽天市場 | Yahoo!ショッピング | au PAY マーケット |
|---|---|---|---|
| 画像URL | 楽天用URL | Yahoo用URLへ変換 | au用URLへ変換 |
| 商品リンク | 楽天商品URL | Yahoo商品URLへ変換 | au商品URLへ変換 |
| カテゴリリンク | 楽天カテゴリURL | 個別変換 | auカテゴリURLへ変換 |
| 使用可能タグ | 楽天仕様 | Yahoo仕様 | au仕様 |
| PC・SP差分 | 個別HTML | 個別対応 | 個別対応 |

すべてを単純な文字列置換として扱うのではなく、用途ごとに処理を分けています。

## 変換ルールの例

実際の接続先や企業固有のURLは公開していませんが、変換処理の考え方は次のようになります。

```python
def convert_image_url(url: str, target_mall: str) -> str:
    """
    元画像URLを、出力先モールに合わせて変換する。

    target_mallを追加することで、
    新しいモール向けの変換処理を拡張できる。
    """
    if target_mall == "yahoo":
        return convert_to_yahoo_url(url)

    if target_mall == "au":
        return convert_to_au_url(url)

    return url
```

変更箇所が分かるように、処理の目的と拡張方法をコメントとして残します。

## 誤変換を減らすための工夫

HTML変換では、対象外の文字列まで置換すると表示崩れやリンク切れにつながります。

そのため、次の点を意識しています。

- 置換対象となるURLやパターンを限定する
- 処理順序を明確にする
- 削除処理と置換処理を分離する
- PC用とスマートフォン用を混在させない
- 変換前の元データを上書きしない
- 変換結果を別ファイルとして保存する
- ログから処理対象を確認できるようにする

処理を追加する際にも、既存の変換へ影響しにくい構成を目指しています。

## 拡張性

新しいモールや変換ルールを追加する場合、既存処理全体を書き換えるのではなく、個別処理を追加できる構成にしています。

想定している拡張例は次のとおりです。

- 対応モールの追加
- 新しいURL変換ルールの追加
- モールごとの禁止タグ追加
- 画像命名規則の変更
- CSV列構成の変更
- GUIからの変換設定
- 変換ルールの設定ファイル化

将来の変更を完全に予測して作り込むのではなく、変更が発生した際に影響範囲を限定できる設計を重視しています。

## 属人化を減らすための工夫

手作業でHTMLを修正する場合、作業者がモールごとの細かな仕様を覚えている必要があります。

このツールでは、その判断を変換ルールとしてプログラムに持たせています。

- 修正箇所を毎回探さなくてよい
- モール仕様をすべて暗記しなくてよい
- 作業者による修正品質の差を減らせる
- 変更ルールをコード上で共有できる
- 担当者が変わっても同じ処理を再現できる

特定の担当者の経験や記憶に依存せず、一定の手順で同じ結果を得られることを目指しています。
