# Mak.K

# Workflow Automation / Business Process Improvement

業務改善・運用設計・自動化をテーマに開発したプロジェクトをまとめています。

EC運営、Web制作、業務自動化の経験を活かし、現場で発生する課題の分析からツール設計・実装・運用まで一貫して取り組んでいます。

Pythonを中心に開発を行いながら、手作業・待機時間・ヒューマンエラー・属人化といった問題の解決や、継続的に運用できる仕組みづくりを重視しています。

単なる作業の自動化ではなく、

- 誰でも利用できること
- 安全に運用できること
- 問題発生時に追跡できること

を意識して設計しています。

現場業務を観察し、作業や情報の流れを整理しながら、より効率的で再現性の高い運用へ改善していくことを得意としています。

# Featured Projects

## 1. [Multi-Mall Sales Aggregation Tool](./projects/sales-aggregation/)

### 概要

複数ECモールの売上CSVを統合し、Excelテンプレートへの転記から親品番集計・SKU集計までを自動化する業務支援ツールです。

モールごとに異なるCSV形式を吸収し、日次・月次集計業務の効率化と標準化を目的として開発しました。

### 対応モール

- 楽天市場
- Yahoo!ショッピング
- au PAY マーケット
- Amazon
- Qoo10
- dショッピング

### Before

- モールごとにCSV形式が異なる
- 手作業による転記ミスが発生する
- 集計作業に時間がかかる
- 特定担当者へ依存しやすい
- ヒューマンエラー発生リスクが高い

### After

- CSVを自動読込
- モールごとの差異を吸収
- Excelテンプレートへ自動転記
- 親品番単位で自動集計
- SKU単位で自動集計
- ログによる処理結果確認

### 実装内容

#### Python

- 集計処理全体の実装

#### pandas

- CSVデータの読込・変換

#### openpyxl

- Excel転記・オートフィル処理

#### YAML

- モールごとの差分設定管理

#### tkinter

- 非エンジニア向けGUI作成

#### logging

- 実行結果・エラー内容の記録

### 設計判断

- 売上データのため完全自動化は採用しない
- 最終確認は人間が行う
- サイレントエラーを防止
- 成功・失敗を問わずログを出力
- CSV取り違えによる事故を防止
- モール追加を考慮してYAML化
- 非エンジニアでも利用できる操作性を重視

### 実際の効果

- 日次集計時間 約25分 → 約3分へ短縮
- 月40時間規模の作業削減見込み
- 転記ミス削減
- 集計作業の標準化
- 属人化の軽減

### Design Decisions

本ツールは単なる集計自動化ではなく、実際の運用現場で継続利用できることを重視して設計しました。

- 売上データを扱うため、完全自動化ではなく最終確認工程を残す構成とした
- モール追加や仕様変更に対応しやすいよう設定をYAML化
- 問題発生時の追跡を容易にするためログ出力機能を実装
- CSV取り違えや設定ミスなどのヒューマンエラーを考慮
- 非エンジニアでも扱えるGUI操作を採用
- 運用担当者が保守しやすい構成を意識して設計

### Lessons Learned

開発当初は作業時間削減を主目的としていましたが、実際には「安全に運用できること」の重要性を強く実感しました。

特に売上データを扱う業務では、完全自動化よりも確認工程を残した半自動化の方が現場に適しているケースが多く、運用性・保守性・確認性を考慮した設計の重要性を学びました。

[詳細な設計・処理内容を見る](./projects/sales-aggregation/)

- ログによる結果確認

- Windowsデスクトップアプリ化

#### WinSCP

- SFTPアップロード処理

#### YAML

- 設定ファイル管理

#### GUI

- 非エンジニア向け操作画面

#### 予約実行機能

- 指定時刻での自動アップロード

#### ログ出力

- 実行結果・エラー内容の記録

### 実際の効果

- 更新待機時間を月約7〜8時間削減
- 年間約90時間の工数削減
- 実行忘れ防止

- コード修正支援を高速化
