# 運用・設計方針（実務メモ）

## 目的
- 更新作業を「迷わず」「ミスなく」回せる状態にする
- 属人化を避け、手順とルールを文章で残す
- 制約（モール仕様・管理画面制限）を前提に設計する

## 基本方針
- まず「変えない部分」を固定する（構造の安定化）
- 次に「変える部分」を集約する（更新箇所を上部・特定ブロックに寄せる）
- ルール化できるものはツール化する（手作業の撲滅）

## 更新しやすい構造の考え方
- 文章・スペック・注意書きなど、差し替え頻度が高い要素を分離する
- テーブル構造は崩れにくさ優先（見た目より運用耐久）
- 例外が出やすい箇所は「例外の扱い」まで決めておく

## 変換ツールの位置づけ
- 「毎回同じ変換」を人間がやるとミスる前提で設計
- 変換ルールはREADMEに書ける粒度まで落とし込む
- 仕様差分（例：PC/SP、モール差）を吸収できる形にする

## 守秘義務への配慮
- クライアント名、実URL、実データは掲載しない
- サンプルはダミー化し、構造・思想・再現性を重視する
